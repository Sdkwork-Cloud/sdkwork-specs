# SDKWork Standards

This directory is the canonical standards entrypoint for SDKWork application and backend development.

All applications, API modules, SDK generator inputs, Java services, Rust local services, frontend shared packages, and agent instructions must reference these files instead of maintaining divergent local standards.

## 4. Minimum Rules

- Agent execution starts from `SOUL.md` and the nearest `AGENTS.md`; agents must not rely on remembered SDKWork rules when a relevant spec exists.
- Every git repository root and every SDKWork application root must have `AGENTS.md` according to `AGENTS_SPEC.md`, with relative links to root `sdkwork-specs`. Tool-compatible roots use files such as `CLAUDE.md`, `GEMINI.md`, and `CODEX.md` only as shims to `AGENTS.md`.
- Code changes start from `CODE_STYLE_SPEC.md` and `NAMING_SPEC.md`; language-specific specs are loaded only when that language or framework is touched.
- Non-trivial behavior, contract, runtime, security, release, or migration work starts from `REQUIREMENTS_SPEC.md`; acceptance criteria and traceability are required before implementation is treated as ready.
- Architecture boundary decisions start from `ARCHITECTURE_DECISION_SPEC.md`; package families, app roots, API/SDK ownership, persistence shape, security posture, runtime topology, and cross-client alignment must not be decided only through implementation drift.
- Human and agent engineering work follows `ENGINEERING_WORKFLOW_SPEC.md`; do not confuse engineering workflow with GitHub Actions workflow packaging in `GITHUB_WORKFLOW_SPEC.md`.
- Code and standards review follows `CODE_REVIEW_SPEC.md`; generated SDK output is reviewed as generated evidence and contract input, not as ordinary authored logic.
- Ready, done, merge, release, and exception decisions follow `QUALITY_GATE_SPEC.md`; evidence bundles are part of the standard, not optional project ceremony.
- Releases follow `RELEASE_SPEC.md`; migrations follow `MIGRATION_SPEC.md`; dependency, build, signing, SBOM, provenance, and attestation evidence follows `SUPPLY_CHAIN_SECURITY_SPEC.md`.
- Rust `src/lib.rs` is a module assembly and re-export boundary. It must not become a catch-all file for handlers, repositories, SQL, DTOs, services, provider clients, and tests.
- Rust crates must be named by responsibility. Business logic uses
  `sdkwork-<domain>-<capability>-service`, SQLx database access uses
  `sdkwork-<domain>-<capability>-repository-sqlx`, HTTP route adapters use
  `sdkwork-router-<capability>-<surface>`, HTTP server processes use
  `sdkwork-<application-code>-api-server`, in-process service containers use `sdkwork-<application-code>-service-host`,
  native hosts use `sdkwork-<application-code>-native-host` or `sdkwork-<application-code>-tauri-host`, background jobs use
  `sdkwork-<domain>-<capability>-worker`, and gateway/proxy processes use
  `sdkwork-<application-code>-gateway`. Generic Rust crate names such as `sdkwork-<application-code>-product`,
  `sdkwork-<application-code>-runtime`, `sdkwork-<application-code>-backend`, `sdkwork-<application-code>-core`,
  `sdkwork-<application-code>-common`, and `sdkwork-<application-code>-manager` are not compliant names.
- Domain naming starts from `DOMAIN_SPEC.md`; do not invent `identity` when the standard domain is `iam`.
- Every git repository root and every SDKWork application root must have a source-controlled `.sdkwork/` workspace with `.sdkwork/skills/` and `.sdkwork/plugins/` according to `SDKWORK_WORKSPACE_SPEC.md`.
- SDKWork application repository root `package.json#scripts` must follow `PNPM_SCRIPT_SPEC.md`: expose `dev`, `build`, `test`, `check`, `verify`, and `clean`; add capability-specific `release:*`, `deploy:*`, `api:*`, `sdk:*`, `db:*`, `gateway:*`, and `topology:*` commands when those capabilities exist; and avoid application-code-prefixed public scripts such as `drive:*`, `im:*`, or `clawrouter:*`.
- Independent git repository roots and SDKWork application roots use the standard project root directory dictionary from `SDKWORK_WORKSPACE_SPEC.md`: `apis/`, `apps/`, `crates/`, `sdks/`, `jobs/`, `tools/`, `plugins/`, `examples/`, `configs/`, `deployments/`, `scripts/`, `docs/`, and `tests/`. ## 6. Future Standards Roadmap

The current standards cover repository/application workspace, engineering lifecycle, agent entrypoints, code structure, contracts, SDKs, runtime, security, tests, release, migration, and documentation. A higher-availability AI software factory should later add these specs when executable rules are ready:

| Future spec | Purpose |
| --- | --- |
| `EVALUATION_SPEC.md` | Agent/code/content success metrics, evaluation datasets, pass/fail scoring, regression tracking, and feedback loops |
| `RELIABILITY_SPEC.md` | High-availability service rules, retry budgets, failover, degradation, idempotency at platform level, and recovery drills |
| `INCIDENT_RESPONSE_SPEC.md` | Incident severity, command structure, customer impact assessment, communication, mitigation, postmortems, and follow-up tracking |
| `VULNERABILITY_MANAGEMENT_SPEC.md` | Vulnerability intake, triage, patch SLAs, advisory publication, dependency CVE handling, and security regression tracking |
| `COMPATIBILITY_SPEC.md` | Cross-version API/SDK/client/runtime compatibility matrix, deprecation windows, and consumer upgrade certification |
| `DATA_LIFECYCLE_SPEC.md` | Data retention, archival, deletion, legal hold, backup restore, and dataset lifecycle evidence beyond privacy policy |
| `FEATURE_FLAG_AND_ROLLOUT_SPEC.md` | Feature flag ownership, targeting, progressive rollout, kill switches, experiment safety, and cleanup deadlines |
| `ENGINEERING_METRICS_SPEC.md` | Delivery, quality, reliability, review, and release metrics with anti-gaming rules and evidence sources |
| `PLATFORM_ENGINEERING_SPEC.md` | Internal platform product ownership, paved-road tooling, self-service contracts, golden paths, and platform support model |
| `AGENT_RUNTIME_SPEC.md` | Long-running agent state, snapshots, event sourcing, task resumption, tool permissions, and audit trails |
| `CONVENTION_DICTIONARY_SPEC.md` | Optional executable discovery rules for the existing convention dictionary without creating a separate dictionary service |

Do not create these as empty policy files. Add them when a repository or platform workflow needs executable rules, validation, and ownership.
