# IAM Directory Template Standard

- Version: 1.0
- Scope: federated organization, department, position, and membership seed templates
- Related: `IAM_RBAC_FEDERATION_SPEC.md`, `IAM_MODULE_MANIFEST_SPEC.md`, `DATABASE_SPEC.md`, `IAM_SPEC.md`

Directory templates describe seedable subject trees. They do not replace runtime directory administration APIs.

## 1. Reference Syntax

| Reference | Meaning |
| --- | --- |
| `$tenant` | Active seed tenant |
| `$org:{code}` | Organization template by `code` |
| `$orgref:{ref}` | Organization template by `ref` |
| `$deptref:{ref}` | Department template by `ref` |
| `$posref:{ref}` | Position template by `ref` |
| `$root` | Primary organization template |

## 2. Organization Template

```json
{
  "ref": "root",
  "code": "root",
  "name": "Root Organization",
  "seedId": "0",
  "organizationKind": "team",
  "tenantBoundaryKind": "exclusive",
  "dataBoundaryKind": "tenant",
  "verificationStatus": "verified",
  "sortOrder": 0
}
```

## 3. Department Template

```json
{
  "ref": "dept.general",
  "code": "general",
  "name": "General",
  "parentRef": "$orgref:root",
  "departmentKind": "general",
  "sortOrder": 10,
  "defaultPositions": ["pos.member"]
}
```

## 4. Position Template

```json
{
  "ref": "pos.member",
  "code": "member",
  "name": "Member",
  "organizationRef": "$orgref:root",
  "departmentRef": "dept.general",
  "positionKind": "member",
  "defaultRoleBindings": ["app_user"]
}
```

## 5. Membership Template

```json
{
  "ref": "bootstrap-owner",
  "membershipKind": "owner",
  "organizationRef": "$orgref:root",
  "isPrimary": true,
  "defaultRoleBindings": ["org_admin", "app_user"]
}
```

## 6. Materialization Rules

- Kernel module seeds tenant `100001` and root org `0` in profile `standard` and above.
- Module templates append departments and positions under discovered organizations.
- Closure tables `iam_organization_closure` and `iam_department_closure` must be generated by the orchestrator, not hand-edited SQL.
- Seeded rows must persist `template_ref`, `module_id`, and `seed_profile` metadata columns when present.

## 7. Seed Profiles

| Profile | Directory content |
| --- | --- |
| `minimal` | tenant + root org only |
| `standard` | minimal + general department + member position |
| `operational` | standard + enabled module templates + optional bootstrap owner membership template |

Seed profile rules:

- `minimal` and `standard` `MUST NOT` create live administrator credentials.
- `operational` may create an organization bootstrap owner only when the deployment manifest declares the owner subject or invite flow. It must not create a default password.
- No seed profile may grant `platform_super_admin` by default. Super-admin activation belongs to platform break-glass governance in `PERMISSION_STANDARD_SPEC.md` and `IAM_RBAC_FEDERATION_SPEC.md`.
- Dev/test demo subjects `MUST` be marked by profile metadata and must not be promoted into production seed data.
- Re-running materialization `MUST` reconcile by stable references and must not duplicate memberships, positions, or role bindings.
