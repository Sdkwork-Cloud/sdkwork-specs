# Media Resource Standard

- Version: 1.0
- Scope: images, video, audio, voice, documents, archives, generated media, Drive-backed files, provider assets, media API DTOs, frontend upload state, RPC messages, database references
- Related: `DRIVE_SPEC.md`, `API_SPEC.md`, `DATABASE_SPEC.md`, `RPC_SPEC.md`, `FRONTEND_SPEC.md`, `SDK_SPEC.md`, `SECURITY_SPEC.md`, `PRIVACY_SPEC.md`
- Canonical location: `specs/MEDIA_RESOURCE_SPEC.md`

This standard defines the SDKWork media resource representation. It exists to stop product modules from scattering bare `url`, `imageUrl`, `videoUrl`, `thumbnailUrl`, `assetUrl`, `coverImage`, `avatarUrl`, and object keys across APIs, databases, frontend state, and events.

`MediaResource` is not a storage lifecycle owner. SDKWork-owned file storage, upload sessions, object metadata, provider configuration, presigned grants, retention, scanning, and deletion are owned by SDKWork Drive and follow `DRIVE_SPEC.md`. `MediaResource` is the canonical structure used by APIs, RPC, frontend services, events, and read models to describe a usable image, video, audio, document, archive, provider asset, or generated artifact.

## 1. Normative Language

The words `MUST`, `MUST NOT`, `SHOULD`, `SHOULD NOT`, and `MAY` are used with RFC-style meaning.

## 2. What Counts As Media

Use `MediaResource` or `MediaResource[]` for fields that represent uploadable, storable, transformable, previewable, moderated, or access-controlled media:

- Product main images, gallery images, detail images, SKU images, videos, manuals, certificates, and downloadable files.
- User avatars, organization logos, app icons, favicons, covers, thumbnails, posters, banners, and QR code images when stored or generated by SDKWork.
- Course videos, lesson attachments, generated images, generated audio, generated video, voice assets, model artifacts, studio catalog assets, skill artifacts, invoices, documents, and exported files.
- AI provider outputs after they enter SDKWork business state.

Bare URL strings remain correct for non-media links and endpoints:

- Official websites, documentation pages, terms, privacy policy, support links, repository links, manifest links, marketplace links, app store links, and download landing pages.
- OAuth authorization URLs, callback URLs, webhook target URLs, return URLs, cashier/payment URLs, QR authorization fallback URLs, and action/navigation URLs.
- API base URLs, provider endpoint URLs, model source URLs, Git repository URLs, Redis/database URLs, and deployment/runtime URLs.
- Third-party adapter DTOs that must preserve provider wire fields such as OpenAI `image_url`, provider `uri`, or provider callback fields. These DTOs must be normalized to `MediaResource` before entering SDKWork-owned business state.

## 3. Canonical API Model

API contracts use lowerCamelCase field names.

```ts
type MediaKind =
  | 'image'
  | 'video'
  | 'audio'
  | 'voice'
  | 'document'
  | 'archive'
  | 'model'
  | 'other';

type MediaSource =
  | 'drive'
  | 'external_url'
  | 'data_url'
  | 'provider_asset'
  | 'generated';

interface MediaResource {
  id?: string;
  kind: MediaKind;
  source: MediaSource;

  url?: string;
  publicUrl?: string;
  uri?: string;

  objectBlobId?: string;

  fileName?: string;
  mimeType?: string;
  sizeBytes?: string;
  checksum?: MediaChecksum;

  width?: number;
  height?: number;
  durationSeconds?: number;

  altText?: string;
  title?: string;
  poster?: MediaResource;
  thumbnails?: MediaResource[];
  variants?: MediaResource[];

  access?: MediaAccess;
  ai?: MediaAiProvenance;
  metadata?: Record<string, unknown>;
}
```

Rules:

- `kind` and `source` are required for every `MediaResource`.
- `id` is required once a media resource is persisted as SDKWork-owned state.
- `url` is a delivery hint, not the system of record. It may be signed, temporary, CDN-derived, or provider-derived.
- SDKWork-owned file resources `MUST` use `source = drive`, `uri = drive://spaces/{spaceId}/nodes/{nodeId}`, and `id = {nodeId}` once the file is persisted in Drive.
- `bucketId`, `objectKey`, and provider object-version fields are not part of standard `MediaResource`. They belong only to Drive internal/admin DTOs or provider-adapter wire DTOs and `MUST NOT` be persisted as business media identity outside Drive.
- `url` is required when `source = external_url`, unless the resource is represented by a provider `uri`.
- `data_url` is allowed only for transient request payloads and must not be persisted as long-term business state.
- `provider_asset` must keep the provider identifier in `uri` or `metadata.providerAssetId` until the provider asset is imported into Drive. After import, any SDKWork-owned file identity is Drive.
- `generated` must include `ai.provenance = generated` or equivalent generation metadata when the source is known. Generated files that enter SDKWork durable state must also have a Drive identity through `uri` or the owning message/content `DriveReference`.
- For message content and other write paths that include both `DriveReference` and `MediaResource`, `resource.uri` `MUST` equal `drive.driveUri`, `resource.id` `MUST` equal `drive.nodeId` when present, and `source = external_url` or `source = data_url` `MUST NOT` be accepted.
- `sizeBytes`, `int64` ids, and other large integer values use strings in public JSON contracts.
- `metadata` is extension space and must not replace first-class fields for kind, source, access, size, MIME type, checksum, dimensions, duration, or AI provenance.

## 4. Supporting API Schemas

OpenAPI components SHOULD define these schemas once per API surface and reuse them.

```yaml
MediaResource:
  type: object
  additionalProperties: false
  required: [kind, source]
  properties:
    id:
      type: string
    kind:
      $ref: '#/components/schemas/MediaKind'
    source:
      $ref: '#/components/schemas/MediaSource'
    url:
      type: [string, "null"]
      format: uri
      description: Delivery URL. May be temporary or signed.
    publicUrl:
      type: [string, "null"]
      format: uri
    uri:
      type: [string, "null"]
    objectBlobId:
      type: [string, "null"]
    fileName:
      type: [string, "null"]
      maxLength: 512
    mimeType:
      type: [string, "null"]
      maxLength: 256
    sizeBytes:
      type: [string, "null"]
      pattern: '^[0-9]+$'
    checksum:
      $ref: '#/components/schemas/MediaChecksum'
    width:
      type: [integer, "null"]
      minimum: 0
    height:
      type: [integer, "null"]
      minimum: 0
    durationSeconds:
      type: [number, "null"]
      minimum: 0
    altText:
      type: [string, "null"]
      maxLength: 512
    title:
      type: [string, "null"]
      maxLength: 255
    poster:
      $ref: '#/components/schemas/MediaResource'
    thumbnails:
      type: array
      items:
        $ref: '#/components/schemas/MediaResource'
    variants:
      type: array
      items:
        $ref: '#/components/schemas/MediaResource'

      $ref: '#/components/schemas/MediaAccess'

      $ref: '#/components/schemas/MediaAiProvenance'

      type: object
      additionalProperties: true
```

If an SDK generator cannot handle recursive schemas reliably, the API MAY replace recursive `poster`, `thumbnails`, and `variants` with non-recursive `MediaVariant` and `MediaRendition` schemas. The semantic fields remain the same.

## 5. Drive, Storage, And Database Contract

Storage facts and business media roles are separate.

SDKWork-owned storage facts belong to Drive tables defined by `DRIVE_SPEC.md`:

- Provider, bucket, region, endpoint, credential reference, path style, lifecycle, and health belong to Drive storage provider contracts.
- Object key, version, etag, checksum, content type, original filename, size, retention, and legal hold belong to Drive storage object metadata.
- Upload sessions, multipart parts, presigned grants, completion attempts, reconciliation, usage, and garbage collection belong to Drive upload/storage service tables.

Business tables must store stable media references, not ephemeral delivery URLs.

Rules:

- A persisted business entity `MUST NOT` store a presigned URL as its system of record.
- A business table `SHOULD` reference `drive_space_id`, `drive_node_id`, `drive_uri`, `media_resource_id`, a Drive-exposed `object_blob_id`, or a domain media relation table rather than storing `object_key` or `url` directly.
- A denormalized `media_resource_snapshot` JSON column is allowed only as a read-model projection. Its schema must be `MediaResource`, and the source reference remains authoritative.
- Object storage object keys `MUST NOT` be scattered in arbitrary business JSON. SDKWork-owned object keys live in Drive only.
- If a media resource can be reused across business objects, ownership and authorization must be represented explicitly. Do not infer cross-tenant access from a shared URL.
- If a media resource is generated by AI, generation provenance should be stored with the media resource or a linked generation task, not only in a prompt log.
- Application domains `MUST NOT` create their own object blob tables, upload session tables, storage provider registries, presign services, or media asset lifecycle stores for SDKWork-owned files. Missing storage capability must be added to Drive.

A domain relation table SHOULD use this pattern:

```text
<domain>_<entity>_media
  id
  tenant_id
  organization_id
  owner_type
  owner_id
  media_role
  drive_space_id
  drive_node_id
  drive_uri
  media_resource_id
  object_blob_id
  resource_snapshot
  resource_hash
  alt_text
  sort_order
  status
  created_at
  updated_at
```

`media_role` is domain-specific and stable. Examples: `main_image`, `gallery_image`, `detail_image`, `sku_image`, `video`, `manual`, `certificate`, `avatar`, `icon`, `logo`, `thumbnail`, `poster`, `attachment`, `invoice_document`, `generated_output`.

### 5.1 Drive-Backed MediaResource Profile

For SDKWork-owned files, the standard media profile is Drive-backed:

```json
{
  "id": "node_01HR6P7ZJQ4A7M2CKA9F0P6R7S",
  "kind": "image",
  "source": "drive",
  "uri": "drive://spaces/space_app_upload_01/nodes/node_01HR6P7ZJQ4A7M2CKA9F0P6R7S",
  "fileName": "demo.png",
  "mimeType": "image/png",
  "sizeBytes": "424242",
  "checksum": {
    "algorithm": "sha256",
    "value": "3f786850e387550fdab836ed7e6dc881de23001b"
  },
  "access": {
    "visibility": "tenant"
  },
  "metadata": {
    "drive": {
      "spaceId": "space_app_upload_01",
      "nodeId": "node_01HR6P7ZJQ4A7M2CKA9F0P6R7S",
      "spaceType": "app_upload",
      "nodeVersion": "1"
    }
  }
}
```

Rules:

- `source = drive` identifies a file whose lifecycle, metadata, permissions, versions, scans, download grants, and retention are owned by SDKWork Drive.
- `uri = drive://spaces/{spaceId}/nodes/{nodeId}` is the stable Drive-backed media identity.
- `id = {nodeId}` is the canonical persisted media id for Drive-backed `MediaResource` snapshots.
- `bucketId`, `objectKey`, provider object-version fields, and presigned `url` values must not be required to attach this resource to a business object.
- `objectBlobId` may be populated only with a Drive-exposed storage object/version id. Do not use raw object keys as a substitute.
- `metadata.drive.spaceId` and `metadata.drive.nodeId` should be present when the public schema permits metadata.

## 6. Merchandise Catalog Profile

Product catalog media must support SPU-level and SKU-level media without inventing string URL fields per use case.

Rules:

- Product main image, gallery image, detail image, product video, document/manual, and SKU image use the product media relation profile.
- SKU image is represented as media with `owner_type = sku` and `media_role = sku_image`, or an equivalent SKU media relation. It is not a standalone `imageUrl` string in any standard SKU payload.
- Product media order uses `sort_order`; the canonical main image is the active `main_image` with the lowest sort order unless the domain defines an explicit primary flag.
- Product media requests may include `altText`, `title`, and `metadata`, but accessibility text must not be embedded only in filenames.
- `commerce_product_media.url` is non-standard for a new application and must be replaced by stable media resource fields such as `resource: MediaResource` or `mediaResourceId + resourceSnapshot`.
- Multi-SKU create/update flows should submit media resources per SKU row so SKU code, barcode, SKU image, price, stock, and specs are validated together.

Recommended product media DTO:

```ts
interface CommerceProductMedia {
  id?: string;
  ownerType: 'spu' | 'sku';
  ownerId?: string;
  mediaRole:
    | 'main_image'
    | 'gallery_image'
    | 'detail_image'
    | 'sku_image'
    | 'video'
    | 'manual'
    | 'certificate';
  resource: MediaResource;
  altText?: string;
  sortOrder?: number;
  status?: 'active' | 'inactive' | 'deleted';
}
```

## 7. Frontend Interaction Contract

Frontend upload and media pickers work with `MediaResource` objects from the first stable boundary.

Rules:

- Upload widgets may hold local `File`, preview object URLs, progress, and errors as UI-only state. These values must not leak into API DTOs.
- SDKWork-owned upload reservation, completion, import, and download grants must go through Drive app SDK/API. Client uploads must use `sdkwork-drive-app-sdk client.uploader.*`; server-side Rust uploads must use the Drive server-side uploader service defined by `DRIVE_SPEC.md`. After Drive completion, the service returns a `MediaResource` or Drive upload result that can be normalized to `MediaResource`.
- Form submit payloads must send stable resource identity and metadata, not browser object URLs or temporary presigned URLs.
- UI previews may use `resource.url`, `resource.publicUrl`, or a local preview while uploading, but persistence must use `id`, Drive `uri`, `objectBlobId`, or provider `uri`.
- Services own normalization from generated SDK DTOs to frontend view models. UI components should not patch raw URL strings into business payloads.
- Media fields must expose accessible labels, alt text where relevant, file type validation, size validation, retry/removal states, and upload progress.

## 8. RPC Contract

RPC contracts use `sdkwork.common.v1.MediaResource` for media payloads and references.

Rules:

- Large media should use Drive references, Drive upload sessions, Drive download grants, or Drive-owned streaming. Do not embed large files in unary `bytes` fields.
- `bytes` is allowed for small thumbnails, signatures, or protocol-specific fragments only when size limits are explicit.
- RPC field names use `snake_case`: `media_resource`, `media_resources`, `object_blob_id`, `mime_type`, `size_bytes`, `duration_seconds`.
- Provider adapter services may expose provider-native fields, but SDKWork business services must normalize to `MediaResource`.

Minimal proto shape:

```proto
message MediaResource {
  string id = 1;
  MediaKind kind = 2;
  MediaSource source = 3;
  string url = 4;
  string public_url = 5;
  string uri = 6;
  string object_blob_id = 7;
  string file_name = 8;
  string mime_type = 9;
  string size_bytes = 10;
  int32 width = 11;
  int32 height = 12;
  double duration_seconds = 13;
  MediaAccess access = 14;
  MediaAiProvenance ai = 15;
  google.protobuf.Struct metadata = 16;
}
```

## 9. AI Provenance

AI-era media must preserve enough provenance for audit, moderation, reruns, and provider portability.

Recommended fields:

```ts
interface MediaAiProvenance {
  provenance?: 'uploaded' | 'generated' | 'edited' | 'imported';
  provider?: string;
  model?: string;
  promptId?: string;
  generationTaskId?: string;
  sourceMediaIds?: string[];
  seed?: string;
  moderationStatus?: 'unknown' | 'pending' | 'approved' | 'rejected' | 'blocked';
  safetyLabels?: string[];
}
```

Rules:

- Generated media that is shown to users, sold, published, embedded in products, or used in automation must retain generation provenance.
- Moderation status must be explicit when media crosses user or tenant boundaries.
- Provider URLs that expire must be ingested into Drive or represented as provider assets with refresh rules.

## 10. Access And Security

```ts
interface MediaAccess {
  visibility: 'private' | 'tenant' | 'organization' | 'public' | 'signed';
  expiresAt?: string;
}

interface MediaChecksum {
  algorithm: 'sha256' | 'md5' | 'etag';
  value: string;
}
```

Rules:

- Signed URLs must be short-lived and treated as delivery tokens.
- Persisted resources must store stable identity and access policy, not only signed URLs.
- Public resources should still retain ownership, checksum, size, MIME type, and lifecycle metadata.
- Logs must not expose secret credentials, raw presign signing material, private bucket credentials, or provider tokens.
- Media metadata may contain personal data. Apply `PRIVACY_SPEC.md` data classification and retention rules.

## 11. Full Standardization

New SDKWork applications must use the latest media contract directly. Do not retain bare media URL fields, dual-write URL mirrors, or introduce transition windows for media URL cleanup when the application has not been published externally.

Standardization steps:

1. Classify each `url` or `*Url` field as media, navigation/link, endpoint/config, callback, payment, or provider adapter wire data.
2. For media fields, replace the field with a natural business-role field whose type is `MediaResource` or `MediaResource[]`.
3. For database schema, replace media URL columns with Drive references, stable media references, relation tables, or `MediaResource` snapshots that are explicitly read-model projections.
4. For API and RPC contracts, remove media URL request/response fields from SDKWork-owned business DTOs.
5. For frontend forms and services, submit Drive references, `MediaResource` identities, and metadata instead of browser object URLs, signed URLs, or raw object keys.
6. For provider adapters, normalize provider-native media URLs or URIs to `MediaResource` before data enters domain services, events, or persistence.

Canonical naming:

- `coverImage` becomes `cover`.
- `avatarUrl` becomes `avatar`.
- `thumbnailUrl` becomes `thumbnail`.
- `posterUrl` becomes `poster`.
- `videoUrl` becomes `video`.
- `audioUrl` becomes `audio`.
- `voiceUrl` becomes `voice`.
- `imageUrl` becomes the domain role, for example `image`, `mainImage`, `galleryImage`, `detailImage`, or `skuImage`.
- `fileUrl` becomes `file`, `attachment`, `document`, or another domain role.
- `assetUrl` becomes `asset`.
- `artifactUrl` becomes `artifact`.
- `documentUrl` becomes `document`.
- `commerce_product_media.url` becomes `resource` plus stable media reference fields.

Avoid generic or redundant names:

- Do not append `Media` to a clear business-role field name when the schema type is already `MediaResource`.
- Do not use `url` for persisted media inside a resource object except as the delivery URL field of `MediaResource`.

## 12. Review Checklist

- [ ] Media fields use `MediaResource` or `MediaResource[]`.
- [ ] Bare URL fields are either allowed non-media URLs or provider adapter wire fields; SDKWork-owned media contracts contain no bare media URL fields.
- [ ] Persisted media has stable identity independent of signed delivery URLs.
- [ ] SDKWork-owned storage lifecycle is implemented by Drive, not by app-local media/object/upload tables.
- [ ] Drive-backed media references use Drive spaces/nodes as the storage lifecycle identity.
- [ ] API, RPC, database, frontend, and SDK contracts agree on field names and semantics.
- [ ] AI-generated media preserves provider, model, task, moderation, and provenance metadata when known.
- [ ] Product catalog media supports SPU and SKU ownership, roles, ordering, and alt text.
- [ ] Frontend upload state does not leak browser object URLs or temporary presigned URLs into business payloads.
- [ ] Security and privacy review covers access policy, retention, logs, signed URL expiry, and tenant isolation.
